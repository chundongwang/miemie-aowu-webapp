## Why Cloudflare for China + Global Access

Most mainstream database-as-a-service options (Supabase, PlanetScale, Neon, Firebase) are blocked or unreliable in mainland China. Vercel is also blocked.

Cloudflare operates in China through a licensed partnership with JD Cloud (JD Cloud Edge Network). Cloudflare Pages, D1, R2, and Workers are all reachable from mainland China without a VPN вЂ” this is the key architectural constraint.

**Alternative if Cloudflare D1 proves insufficient** (e.g., complex queries, larger data): swap D1 for a PostgreSQL instance on Alibaba Cloud Hong Kong region, which is accessible from both mainland China and internationally. The API layer stays the same; only the DB driver changes.

## Database Schema

### `users`

```sql
CREATE TABLE users (
  id          TEXT PRIMARY KEY,           -- nanoid
  username    TEXT NOT NULL UNIQUE,
  email       TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  display_name TEXT NOT NULL,
  created_at  INTEGER NOT NULL            -- unix timestamp
);
```

### `lists`

```sql
CREATE TABLE lists (
  id              TEXT PRIMARY KEY,
  owner_id        TEXT NOT NULL REFERENCES users(id),
  recipient_id    TEXT REFERENCES users(id),  -- null until shared
  title           TEXT NOT NULL,
  emoji           TEXT NOT NULL DEFAULT 'рџ“‹',
  category        TEXT NOT NULL DEFAULT 'custom',  -- 'coffee', 'music', 'restaurant', 'book', 'movie', 'custom'
  secondary_label TEXT,                -- e.g. "Artist", "Address" вЂ” overrides category default
  created_at      INTEGER NOT NULL,
  updated_at      INTEGER NOT NULL
);
```

### `items`

```sql
CREATE TABLE items (
  id            TEXT PRIMARY KEY,
  list_id       TEXT NOT NULL REFERENCES lists(id) ON DELETE CASCADE,
  name          TEXT NOT NULL,
  secondary     TEXT,                  -- address, artist, author, etc.
  reason        TEXT,
  status        TEXT NOT NULL DEFAULT 'unseen',  -- 'unseen' | 'saved' | 'done'
  position      INTEGER NOT NULL DEFAULT 0,       -- for manual reordering
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL
);
```

### `item_photos`

```sql
CREATE TABLE item_photos (
  id        TEXT PRIMARY KEY,
  item_id   TEXT NOT NULL REFERENCES items(id) ON DELETE CASCADE,
  r2_key    TEXT NOT NULL,            -- R2 object key
  position  INTEGER NOT NULL DEFAULT 0
);
```

### `invite_links`

```sql
CREATE TABLE invite_links (
  token       TEXT PRIMARY KEY,        -- random 24-char token
  list_id     TEXT NOT NULL REFERENCES lists(id) ON DELETE CASCADE,
  created_at  INTEGER NOT NULL,
  expires_at  INTEGER,                 -- null = no expiry
  used_by     TEXT REFERENCES users(id)
);
```

## API Routes

All routes are under `/api`. Auth is via a short-lived JWT in an `HttpOnly` cookie.

### Auth

| Method | Path | Description |
|---|---|---|
| POST | `/api/auth/register` | Create account |
| POST | `/api/auth/login` | Get JWT cookie |
| POST | `/api/auth/logout` | Clear cookie |
| GET | `/api/auth/me` | Current user |

### Lists

| Method | Path | Description |
|---|---|---|
| GET | `/api/lists` | Lists owned or shared with me |
| POST | `/api/lists` | Create a list |
| GET | `/api/lists/[id]` | Get list + items |
| PATCH | `/api/lists/[id]` | Update title/emoji/label |
| DELETE | `/api/lists/[id]` | Delete list (owner only) |
| POST | `/api/lists/[id]/share` | Share with user by username |
| DELETE | `/api/lists/[id]/share` | Revoke sharing |
| POST | `/api/lists/[id]/invite` | Generate invite link |

### Items

| Method | Path | Description |
|---|---|---|
| POST | `/api/lists/[id]/items` | Add item |
| PATCH | `/api/items/[id]` | Update item fields |
| PATCH | `/api/items/[id]/status` | Update status (unseen/saved/done) |
| DELETE | `/api/items/[id]` | Delete item |
| POST | `/api/items/[id]/photos` | Upload photo (returns R2 URL) |
| DELETE | `/api/items/[id]/photos/[photoId]` | Remove photo |

### Users

| Method | Path | Description |
|---|---|---|
| GET | `/api/users/search?q=` | Find user by username (for sharing) |

## Auth Flow

1. User submits email + password
2. Server checks `users` table, verifies bcrypt hash
3. Signs a JWT (`{ sub: userId, exp: +7d }`) with a secret stored in Cloudflare Workers env
4. Sets `Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict`
5. Client never sees the raw JWT вЂ” all subsequent API calls send the cookie automatically

Password reset v1: out of scope (manual support).

---

## Photo Upload Flow

1. Client calls `POST /api/items/[id]/photos` with the file (multipart)
2. API route streams the file to R2 under key `items/{itemId}/{photoId}`
3. Returns a signed R2 URL (or a proxy URL through the API) for display
4. `item_photos` row is created

Photos are served through a Cloudflare R2 public bucket or a Worker that adds auth checks.
